# 送货单流程修正

## 背景

从送货单 #1（人民医院）发现三个问题：
1. 送货单创建时写入 `transaction (category=sale)`，和零售页面的 sale 混淆
2. 创建送货单不做库存校验，导致库存可为负
3. 换货 exchange 产生多余的 transaction，且金额变化后旧 transaction 对不上

## 设计

### 一、transaction 类别

| 类别 | 来源 | 含义 |
|------|------|------|
| `sale` | 零售页面 | 零售收入 |
| `delivery` | 送货单创建 | 应收发生 |
| `delivery_cancel` | 换货冲抵 | 冲抵旧的应收金额 |
| `payment` | 结算还款 | 还款到账 |

命名跟着 `purchase_cancel` 的惯例。

### 二、exchange 改为 cancel + new

换货时不再直接 UPDATE 旧 transaction，而是追加一对新记录：

- return_items：写 stock_movement（direction=in, reason=return）
- new_items：写 stock_movement（direction=out, reason=sale）
- 产生一条 `delivery_cancel` transaction（amount = -旧金额），冲抵旧的 delivery
- 产生一条新的 `delivery` transaction（amount = 新金额）
- 更新 `delivery.total_amount` 为新金额

### 三、应收公式

```
应收 = delivery + delivery_cancel - payment - subscription
```

```sql
SUM(CASE
  WHEN category IN ('delivery', 'delivery_cancel') THEN amount
  WHEN category IN ('payment', 'subscription') THEN -amount
  ELSE 0
END)
```

改动位置：`TransactionRepository.get_ar_by_customer` 和 `get_receivables`

### 四、送货库存校验

`create_delivery` 写入前检查每个品项的库存是否 ≥ 出货量，不足则 `raise ValueError`。exchange 的 new_items 同理。

### 五、批量结算

新增 `POST /api/settlements/batch`：

```json
{
  "customer_id": 1,
  "items": [
    {"delivery_id": 1, "amount": 300},
    {"delivery_id": 3, "amount": 500}
  ]
}
```

批量生成 payment transaction。

### 六、已有数据修正

手动将数据库中送货单 #1 的 transaction category 从 `"sale"` 改为 `"delivery"`。
