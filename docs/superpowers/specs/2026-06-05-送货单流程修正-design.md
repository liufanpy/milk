# 送货单流程修正

## 背景

从送货单 #1（人民医院）发现多个问题：
1. 送货单创建时 `stock_movement.reason` 和 `transaction.category` 都写 `"sale"`，和零售混淆
2. 创建送货单不做库存校验，库存可为负
3. 换货 exchange 产生多余的 transaction，金额变化后旧记录对不上

## 设计

### 一、命名统一

stock_movement 和 transaction 统一用 `"delivery"`：

| 来源 | stock_movement.reason | transaction.category |
|------|----------------------|---------------------|
| 零售 | `sale` | `sale` |
| 送货单 | `delivery` | `delivery` |
| 退货 | `return` | — |
| 换货冲抵 | — | `delivery_cancel` |
| 结算 | — | `payment` |
| 订奶 | `sale` | `subscription` |

### 二、exchange 改为 cancel + new

换货不再直接改旧 transaction：

- return_items：stock_movement（direction=in, reason=return, delivery_id）
- new_items：stock_movement（direction=out, reason=delivery, delivery_id）
- 追加 `delivery_cancel` transaction（amount = -旧金额）
- 追加新的 `delivery` transaction（amount = 新金额）
- 更新 `delivery.total_amount`

### 三、应收公式

```
应收 = delivery + delivery_cancel - payment - subscription
```

```sql
SUM(CASE
  WHEN category IN ('delivery', 'delivery_cancel') THEN amount
  WHEN category IN ('payment', 'subscription') THEN -amount
  ELSE 0
END)
```

### 四、库存校验

`create_delivery` 和 exchange 的 `new_items` 扣库前校验库存 ≥ 出货量，不足则拒绝。

### 五、批量结算

新增 `POST /api/settlements/batch`：

```json
{
  "customer_id": 1,
  "items": [
    {"delivery_id": 1, "amount": 300},
    {"delivery_id": 3, "amount": 500}
  ]
}
```

### 六、已有数据修正

- 送货单 #1 的 `stock_movement.reason` 从 `"sale"` 改为 `"delivery"`
- 送货单 #1 的 `transaction.category` 从 `"sale"` 改为 `"delivery"`
