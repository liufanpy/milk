# 进货单表设计

## 目标

新增 `purchase_orders` 表作为进货单头，将当前零散的 `stock_movements` 进货记录组织为结构化进货单。

## 数据库变更

### 新增表：`purchase_orders`

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | INTEGER | PK, AUTOINCREMENT | 主键 |
| order_number | VARCHAR(20) | NOT NULL, UNIQUE | 单号，格式 PO-YYYYMMDD-NNN |
| supplier_id | INTEGER | FK → suppliers.id, NOT NULL | 供应商 |
| purchase_date | DATE | NOT NULL | 进货日期 |
| total_amount | FLOAT | NOT NULL, DEFAULT 0 | 总金额（冗余，便于列表展示） |
| note | VARCHAR(500) | DEFAULT '' | 备注 |
| status | VARCHAR(20) | NOT NULL, DEFAULT 'draft' | draft / confirmed / cancelled |
| created_at | DATETIME | NOT NULL | 录入时间 |
| updated_at | DATETIME | NOT NULL | 更新时间 |

### 已有表新增字段

- `stock_movements` 新增 `purchase_order_id` INTEGER FK，可空
- `transactions` 新增 `purchase_order_id` INTEGER FK，可空

## 业务逻辑

### 状态流转

```
draft → confirmed（入库）
draft → cancelled（草稿作废）
confirmed → cancelled（撤销入库，反向冲抵库存）
```

### 操作

- **创建草稿**: 仅写 `purchase_orders`，不写 `stock_movements` / `transactions`
- **确认入库**: 批量写 `stock_movements`（direction=in, reason=purchase）+ 一条 `transaction`（category=purchase），`purchase_orders.status` 改为 confirmed
- **撤销**: 追加反向 `stock_movement`（direction=out, reason=purchase_cancel）+ 反向 `transaction`，status 改为 cancelled
- **淘汰 `purchase_date`**: 已通过 `purchase_orders` 保存，无需在 `stock_movements.created_at` 中推断

### 单号生成

格式 `PO-YYYYMMDD-NNN`，当日自增序号（如当天已有 5 条，新单为 PO-20260605-006）。确认时生成还是创建草稿时生成待定，建议创建时即生成（即使草稿也占号，避免跳号混乱）。

## 前端页面

### 进货列表页

- 展示 `purchase_orders` 列表，字段：单号、供应商、进货日期、总金额、状态
- 每行点击进入明细弹窗（展示该单所有品项 product/qty/unit_cost/shelf）

### 新建进货单

- 保持现有表单结构（供应商 + 多品项行 + 备注）
- 新增「保存草稿」按钮和「确认入库」按钮
- 草稿可后续编辑/确认/作废

### CSV 导入

- 导入确认时直接创建 confirmed 状态的进货单（不改默认行为，之后可配置）

## 后续拆分扩展

当前明细复用 `stock_movements`（通过 `purchase_order_id` 关联）。未来如需独立 `purchase_order_items` 表：

1. 一条 SQL 从 `stock_movements` 迁移数据
2. 修改 PurchaseService 读写明细的逻辑，指向新表
3. API 返回结构调整

工作量可控，不是技术债务。
